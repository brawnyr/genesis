# UI Redesign: Canvas, Pads & Engine Terminal

**Date:** 2026-03-10
**Status:** Approved

## Overview

Complete UI layout overhaul. Replace the current left-panel channel list + right-panel Genesis Terminal with a unified full-width canvas, horizontal pad strip, and real-time engine event interpreter. The LLM-powered Genesis Terminal is removed — replaced by a deterministic engine event stream that interprets what's actually happening in the audio process.

## Layout (top to bottom)

### 1. Transport Bar (top)

Single horizontal bar, full width. Contains:
- Play/stop icon (▶/■) with state color (blue when playing)
- BPM (bold white when playing, dim when idle)
- Bar count in brackets: `[4] bars`
- Metronome toggle: `♩ on/off`
- Beat counter (only when playing): `beat 3`
- GOD capture status: `○ GOD` (idle) / `◉ GOD — armed` (orange) / `◉ GOD — recording` (orange, pulsing)
- Master volume: `master 85%`
- Loop progress bar (small, ~80px, inline)

### 2. The Canvas (middle, flex-grows to fill)

One unified zone. Two layers composited:

**Background layer — Pad Visuals:**
- Each pad owns a vertical column (12.5% width, aligned to its pad below)
- When a pad fires: gradient column rises from the bottom, height = intensity, fades over time
- Short hits (kick, snare): quick burst, fast decay
- Sustained/looping samples: persistent glow, height tracks playback progress
- Muted pads: no visual activity
- Color: Claude blue (`#6283e2`) for normal, orange (`#da7b4a`) for long loops or hot signals
- Visuals spring upward from the pad strip below

**Foreground layer — Engine Terminal:**
- Scrolling monospace text, newest at bottom, oldest fades toward top (opacity gradient)
- Text overlays on top of the visual columns — they coexist in the same space
- Lines are generated by the **Engine Event Interpreter** (see section below)

**ASCII GOD Title:**
- Centered in the canvas
- **Idle state:** dim charcoal (`#2a2825`), static
- **Playing state:** Claude blue (`#6283e2`) with subtle glow (`text-shadow`), alive
- The letter D must be clearly distinct from O — choose an ASCII art font where all three letters are unambiguous
- Z-order (back to front): 1) pad visual columns, 2) ASCII title, 3) terminal text
- Subtitle: `GENESIS ON DISK` in small caps below the title (idle only)
- `press SPACE to begin` hint below subtitle (idle only)

### 3. Pad Strip (bottom, above hotkeys)

8 pads laid out horizontally, equal width (except active pad slightly wider).

**Each pad shows:**
- Pad number (1-8), bold and colored when active/triggered
- Sample name (short, e.g. `KICK`, `SNARE`)
- Signal meter (thin 3px bar showing current level)
- Top border colored by state: blue (active), orange (hot/sustaining), dim grey (muted)

**Active pad (selected via A/D navigation):**
- Slightly wider than other pads (`flex: 1.4` vs `flex: 1`)
- Glow/box-shadow to distinguish
- Shows CC parameter values inline:
  - `vol 72%`
  - `pan C` (or `L30`, `R45`, etc.)
  - `HP 20` (Hz, dim when at 20 = no effect)
  - `LP 4.2k` (Hz, highlighted orange when filter is engaged, dim at 20k = no effect)
- CC values update in real time as MiniLab knobs are turned

**Muted pads:**
- Dimmed (opacity 0.5)
- Border color grey
- No signal meter activity

**Trigger animation:**
- On MIDI note-on: pad number flashes orange briefly, background pulses

### 4. Hotkeys Strip (very bottom)

Full-width bar with all keyboard shortcuts:

| Key | Action |
|-----|--------|
| `SPC` | play/stop |
| `G` | toggle GOD capture |
| `A` | select pad left |
| `D` | select pad right |
| `W` | toggle mute on active pad |
| `S` | TBD (reserved) |
| `M` | toggle metronome |
| `↑` | BPM + 1 |
| `↓` | BPM - 1 |
| `[` | fewer bars |
| `]` | more bars |
| `-` | master volume down |
| `+` | master volume up |
| `Z` | undo last clear |
| `C` | clear active pad layer |
| `T` | open setup (sample loader) |
| `ESC` | stop playback |
| `?` | show help overlay |

**WASD replaces 1-8 for mute.** Muting is now `W` on the active pad (navigated with A/D). The old 1-8 mute shortcuts are removed.

### 5. Idle vs Playing States

**Idle:**
- Transport bar dim (grey text)
- Canvas shows only the ASCII GOD title (charcoal) + subtitle + "press SPACE"
- No visual columns, no terminal text
- Pads dim, no signal activity
- Everything quiet and still

**Playing:**
- Transport bar alive (white text, blue accents, beat counter visible)
- ASCII GOD title glows blue
- Visual columns react to pad activity
- Terminal text streams engine events
- Active pad glows, CC values visible
- Loop progress bar animates

## Engine Event Interpreter

The core new system. Replaces the LLM-powered Genesis Terminal with deterministic, real-time interpretation of engine state.

### Architecture

```
processBlock() + layer state + transport
        ↓
  EngineEventStream (structured typed events)
        ↓                    ↓
  Terminal Formatter    Visual Renderer
  (text lines)         (pad glow/columns)
        ↓                    ↓
       The Canvas (composited)
```

### Event Types

The interpreter watches `processBlock` output and engine state changes, producing typed events:

**Hit events:**
- `kick .1s — vel 112` (sample name, duration, velocity)
- `snare .08s — vel 127 — hard hit` (flag hard hits at high velocity)

**Layer summary events** (periodic, not per-hit):
- `hats (4 hits, tight vel 88-110)` (hit count within current loop, velocity spread)
- `bass loop int 14.2s on top of kicks (7 hits, varying vel)` (long sample duration, what it's layered with, hit count)

**CC change events:**
- `pad 1 vol → 72%` (CC 14)
- `pad 1 LP → 4.2kHz` (CC 17)
- Only emitted when a CC value actually changes

**Transport events:**
- `▶ loop start — 4 bars @ 120bpm (8.0s)`
- `loop boundary — wrap`
- `bpm → 130`

**Mute events:**
- `pad 4 muted ○`
- `pad 4 unmuted ●`

**Capture events:**
- `capture armed — next loop boundary`
- `capture recording ◉`
- `capture saved — GOD_20260310_143022.wav`

**Clear/undo events:**
- `pad 3 cleared`
- `pad 3 undo — 12 hits restored`

### Formatting Rules

- Short samples (< 0.5s): show duration in seconds, velocity
- Long samples (> loop length): flag as `loop int {duration}`, note what else is playing
- Velocity descriptors: `vel 0-40` = soft, `41-90` = normal (just show number), `91-127` = hard hit
- Hit summaries: aggregate per-layer within a loop, show count + velocity range descriptor (tight = small spread, varying = large spread)
- CC changes: show pad number, parameter name, new value, only when value changes
- Newest lines = full opacity. Oldest lines fade toward 0.3 opacity.
- Max ~30 visible lines before oldest scroll off

### Data Sources

The interpreter consumes two distinct data feeds from GodEngine:

1. **Per-hit data from `pendingHits`** — the audio thread already accumulates `(padIndex, position, velocity)` tuples in `pendingHits` and flushes them to the UI thread at ~30Hz. The interpreter consumes these directly for hit events. Each hit carries the pad's sample reference (for duration lookup via `PadBank`).

2. **State diffs** — for CC changes, mute toggles, transport changes, and capture state. The interpreter snapshots layer/transport/capture state each tick and diffs against previous to detect changes.

This hybrid approach gives per-hit granularity (velocity, timing) while also catching state changes that don't originate from hits.

### Layer Summary Timing

Layer summaries are emitted **once per loop boundary** (when transport wraps). At wrap, the interpreter counts hits per layer in the completed loop and emits a summary line with hit count + velocity spread.

### Formatting Details

**Frequency display:** Below 1000 Hz → integer Hz (e.g., `340Hz`). At/above 1000 Hz → one decimal with `k` suffix (e.g., `4.2kHz`). At extremes: `20Hz` = off (HP), `20kHz` = off (LP).

**Pan display:** 0.5 = `C` (center). Below 0.5 = `L` + percentage (e.g., `L30`). Above 0.5 = `R` + percentage (e.g., `R45`).

### Implementation

New file: `GOD/GOD/Engine/EngineEventInterpreter.swift`

- Consumes `pendingHits` array and state diffs at UI throttle rate (~30Hz)
- Holds reference to `PadBank` for sample duration lookups
- Produces `EngineEvent` structs (enum with associated values)
- Terminal formatter converts events to attributed strings
- Visual renderer converts events to animation parameters per pad

## Visual Renderer

Drives the pad visual columns in the canvas background layer.

### Per-Pad State

Each pad maintains visual state:
- `intensity: Float` (0-1, current visual energy)
- `decayRate: Float` (how fast intensity drops)
  - Short hits (< 0.5s): decay = 0.92 per frame (~30Hz tick → fades in ~0.4s)
  - Sustained samples: decay = 0.98 per frame (~fades in ~1.5s after voice ends)
- `color: Color` (blue for normal, orange for long loops / high energy)

### Behavior

- **On hit:** intensity spikes to velocity-scaled value, decay begins
- **Sustained playback:** intensity holds at a base level while voice is active
- **On mute:** intensity drops to 0 immediately
- **Visual column height** = intensity * max canvas height
- **Gradient:** solid at bottom, fades to transparent at top

### Rendering

SwiftUI Canvas or overlay views positioned absolutely, aligned to pad columns below. Background layer (behind terminal text).

## Keyboard Changes

### New bindings
- `A` — select previous pad (wraps 1→8)
- `D` — select next pad (wraps 8→1)
- `W` — toggle mute on active pad
- `C` — clear active pad layer

### Changed bindings
- `T` — now opens setup (was terminal toggle — terminal is always visible now)
- `S` — removed (was setup, now reserved for future use)
- `1-8` — removed (muting is now W on active pad)

### Unchanged
- `SPC`, `G`, `M`, `↑`, `↓`, `[`, `]`, `-`, `+`, `Z`, `ESC`, `?`

### Reserved
- `S` — TBD

### Key codes (macOS virtual)
- `A` = keyCode 0, `D` = keyCode 2, `W` = keyCode 13, `C` = keyCode 8, `T` = keyCode 17
